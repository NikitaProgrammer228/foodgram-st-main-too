# Подготовка к защите проекта Foodgram

Краткая инструкция по пунктам от преподавателя: чек-лист, спецификация API, коллекция Postman, тестовые данные.

---

## 1. Чек-лист

Пункты чек-листа берутся из задания/методички вашего курса. Проверьте, что в проекте реализовано всё из задания (регистрация, рецепты, избранное, подписки, список покупок, скачивание списка и т.д.). На защите нужно будет показать выполнение этих пунктов.

---

## 2. Спецификация API

- **Файл в репозитории:** `docs/openapi-schema.yml` (OpenAPI 3.0).
- **Интерактивная документация в браузере:** после запуска проекта откройте  
  **http://localhost/api/docs/** (при работе через Docker) или  
  **http://127.0.0.1:8000/api/docs/** (если запускаете только backend).

Там перечислены все эндпоинты, форматы запросов/ответов и коды ответов. Имеет смысл перед защитой пробежаться по разделам: Пользователи, Рецепты, Ингредиенты, Избранное, Подписки, Список покупок.

## 3. Коллекция Postman — главная часть защиты

### 3.1 Где лежит коллекция

- Файл: **`postman_collection/foodgram.postman_collection.json`**
- Подробности: **`postman_collection/README.md`**

### 3.2 Загрузка в Postman

1. Откройте Postman.
2. **File → Import** и выберите `foodgram.postman_collection.json` (или перетащите файл в окно).
3. Коллекция появится в левой панели (Collections).

### 3.3 Переменная baseUrl

От того, как вы запускаете проект, зависит адрес API:

| Как запускаете проект | baseUrl в коллекции |
|-----------------------|----------------------|
| **Docker Compose** (полный стек: `cd infra && docker-compose up -d`) | `http://localhost` |
| **Только backend** (`python manage.py runserver`) | `http://127.0.0.1:8000` (значение по умолчанию в коллекции) |

Как поменять:

1. В Postman откройте коллекцию **diploma** (клик по названию).
2. Вкладка **Variables**.
3. В колонке **Current value** для переменной **baseUrl** укажите нужный адрес (например, `http://localhost` для Docker).
4. Сохраните (Ctrl+S).

### 3.4 Запуск коллекции

1. Правый клик по коллекции → **Run collection** (или кнопка Run).
2. В правой панели включите **Persist responses for a session**, чтобы видеть ответы после прогона.
3. Нажмите **Run diploma** (или название коллекции).
4. Просмотр упавших тестов: вкладка **Failed**.

### 3.5 Что нужно уметь на защите

- Объяснять, что делает каждый запрос (регистрация, логин, создание рецепта, избранное, подписки, список покупок и т.д.).
- Уметь **изменить** запрос: поменять тело (JSON), параметры, заголовки и понять, какой ответ ожидается (200, 201, 400, 401, 403, 404).
- Понимать авторизацию: после **get_token** токен подставляется в заголовок **Authorization: Token &lt;token&gt;** для запросов от имени пользователя.

## 4. Тестовые данные и запуск проекта

### 4.1 Локальный запуск через Docker (рекомендуется для защиты)

Из корня проекта:

```bash
# .env уже должен быть в backend/foodgram_backend/
cd infra
docker-compose up -d --build
docker-compose exec backend python manage.py migrate
```

**Ингредиенты** (обязательно минимум 2 для коллекции Postman):

В Docker папка `data` монтируется в контейнер как `/app/fixtures`. В `data/` есть **`ingredients.json`** с двумя ингредиентами. Загрузите их:

```bash
docker-compose exec backend python manage.py load_ingredients ingredients.json
```

Если используете свой файл с ингредиентами в формате  
`[{"name": "Название", "measurement_unit": "г"}]`, положите его в `data/` и подставьте его имя в команду.

**Опционально — полные тестовые данные** (если есть файл фикстур):

```bash
docker-compose exec backend python manage.py loaddata /app/fixtures/foodgram_data.json
```

(Путь может отличаться в зависимости от того, куда в контейнере смонтирована папка с данными.)

**Суперпользователь** (для входа в админку):

```bash
docker-compose exec backend python manage.py createsuperuser
```

После этого:

- Сайт: **http://localhost/**
- API: **http://localhost/api/**
- Документация API: **http://localhost/api/docs/**
- В Postman выставляйте **baseUrl** = `http://localhost`.

### 4.2 Только backend (без Docker)

Если запускаете только backend для проверки API в Postman:

1. Активируйте виртуальное окружение, установите зависимости.
2. В `backend/foodgram_backend/foodgram_backend/settings.py` можно временно переключиться на SQLite и `DEBUG = True` (как в README коллекции).
3. Выполните миграции, загрузите минимум 2 ингредиента (через админку или свою команду/фикстуры).
4. Запустите: `python manage.py runserver`.
5. В Postman **baseUrl** оставьте **http://127.0.0.1:8000**.

### 4.3 Пользователи для коллекции Postman

Коллекция **сама создаёт трёх пользователей** и получает для них токены:

- 1-й: email и username из переменных `email`, `username` (в коллекции по умолчанию — vasya.ivanov / vivanov@yandex.ru).
- 2-й: `secondUserEmail`, `secondUserUsername`.
- 3-й: `thirdUserEmail`, `thirdUserUsername`.

Пароль у всех один — переменная **password** (по умолчанию `MySecretPas$word`).

Дополнительно можно заранее создать тестовых пользователей через:

- **Админку:** http://localhost/admin/ (или http://127.0.0.1:8000/admin/).
- **API:** POST **/api/users/** с полями `email`, `username`, `first_name`, `last_name`, `password`.

- API успешно обрабатывает запросы из коллекции Postman (коллекция запускается без критичных падений или вы их можете объяснить и исправить).
- Вы можете продемонстрировать запросы в Postman и при необходимости изменить их (URL, метод, тело, заголовки).
- Проект можно развернуть локально (достаточно локального запуска на своём компьютере).
